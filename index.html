Web Dev Concepts

Web Servers

The HTML file "index.html"--sometimes "default.html"--is the conventional homepage used for web servers. This file is called the designated homepage which is the directory where the web server will take you when the "/" request is made. This convention can transcend page format where, in addition to index.html, you will see other languages use the same covention such as index.js and index.php used as well.

When comparing the content in the Web Dev/Inspector Network tab between the local and web server noticeable differences are found. You will first notice that after clicking on the "index.html" file, the server has a substantial amount of information under the Request and Respond Header that the local file does not have such as age, connection, content-decoding content-length, etc. Most notably, you will see that the server has a favicon.ico file with the OSU icon displaying on the tab.

When one inspects the files in the Web Dev/Inspector Network tab, you will see that the HTML response code is 200, as expected, as that code lets us know the file request was successfully retrieved. However, the same cannot be said for the CSS and JS files. Despite the source files being referenced in the head-section above, the files do not exist, thus, the 404 code is returned which means that our URL attempted to search for the files, but nothing was found.

The following is a component breakdown of this file's URL: https://web.engr.oregonstate.edu/~gochezjo/a2-gochezjo. Our scheme, which identifies the protocol used to send the request, is: "https". The subdomain, which is an additional directory appended by the host name, to help narrow down the search, is : "web.engr." The host name, which is the name of the host/server name of the machine which carries the desired content which we are attempting to connect to, is: "oregonstate.edu". Finally, our resource path, which is the path through the directory and folders that gets us to the desired file, is "~gochezjo/a2-gochezjo". In addition, while our URL does not include the root file, it is implied: "index.html".

Frontend Design

The concept of frontend design frontend design focuses around the user's experience in using any given program, application, webpage, etc. One of the pivotal features of frontend design is the visual and interactive design of a page, also known as the graphical user-interface (GUI). These designs can include icons or images decorated with fonts and colors that help the user assimilate themselves to the page and quickly grasps its functionalities, and of course, enjoy the experience. Frontend design can be further broken down into what engineers have called the five “E”'s of Usability.

Five “E” s of Usability

Effective
- an effective page is one where the overall design yields the correct functionality that leads one to receive the correct information, destination, and overall expectation.
Efficiency
- an efficient page isn't just one that works, but it's one which is designed to help the user get to their desired goal in the least amount of steps and quickest time possible.
Ease of Navigation
- arguably one of the most important features is the ability to design a page with the most user-friendly interface. This page's accessibility would make even a newcomer's first experience in navigating the path's of the page to their destination as seamless as possible and easy to repeat.
Error-free
- an error-free page is one where the designer has figured out common logical and technical roadblocks and issues that users may run into and designed the page to help the user avoid them. Edge cases can be real thorns on the developer's side that could be the cause of bad feedback.
Enjoyable
- finally, whether a developer is dealing with a competitive market or with a community that requires certain services, the final touch to any page and arguably the factor that will make users come back is how much they enjoyed the experience. This includes the effectiveness of the previous four “E” s and any additional visual and interactive designs that makes the experience memorable and even desirable.

Purpose of Page Layout Tags

<Header>

The header tag represents the page banner that introduces the page and its most basic biographical information such as the name, purpose, and slogan, etc. The convention is to keep this as small as possible and appear throughout every internal page.

<nav>

Short for navigation, this tag takes us from the current page to the page link attached to this tag. It is common for these tags to take us to the page's main menu, contacts, etc.

<main>

This tag will be the block that holds the primary information and content of the page. Because of this vital role it is a distinct tag to the section tag below.

<section>

The content of a page may be extensive, so a way to organize the content thematically is through the section tag. Typically we will have each section started off with a headline <h1> describing the section.

<article>

When the <section> tag has been deployed and more content related to this specific section exists, it is good practice to organize this content with article tags within the section. Just as the section tag, the article tag should be started off with a headline, however, of lower grade: <h2>. If multiple articles exist, its a great heuristic to have these tags labeled with an ID selector (#topic) to allow a nav tab to link to that particular article.

<aside>

Used as a supplement to an article, the aside tag will typically float on the sides with a distinct style but relatable content.

<figure>

Similar to the aside tag, the figure tag will supplement an article with some related media, whether an image, gif, etc. This supplemental media will be tagged with <img> and <figcaption> tags, with the name of the tags describing their purpose all within the figure tag.

<blockquote>

While text is usually displayed with a <p> such as the tags that are currently displaying this paragraph we are reading, when the text is extensive it is recommended to use the blockquote tag. It is also visually different to emphasize its disctinct length which comes typically with a purpose in mind from the standpoint of the page. This tag will include two other tags: quote <q> and citation <cite>, which go within the blockquote tag.

<footer>

Typically the last bit of content one sees at the very end of the page. The convential purpose is to display the copyright, year of publication and owner's name. We will also see legal and contact information with links to relevant and important pages. One will need to use ASCII entities to display particular symbols not found in a traditional keyboard.

<div>

The div tag is primarily used as a placeholder, whether for stand alone information or for visual spacing of content. It has no style or dimension, it is truly just a placeholder when no other appropriate tag exists.

How anchors link to:

  1. External Content: one must use the external content's URL within the anchor tag <a>. We must also include a description of the URL within the anchor tag which will be displayed with blue font to the user to point out it is a link. The following is an example of how this looks in HTML (note: words in bold are the only things the user needs to modify): <a href="https://developer.mozilla.org/en-US/"> MDN's Developer Website</a>
  2. Internal Content: the tag where the desired content is at--where we plan to navigate to--must be given an ID. The user then uses an anchor tag identically to that appearing above in external content, where the only difference lies in replacing the URL with a "#" followed directly by the ID found in the desired tag. Here is an example of the both the tag where the ID and thus location is created followed by the anchor tag that takes us there (note: words in bold are the only things the user needs to modify): <article id="topic">The content for this topic...</article> <a href="#topic" >Topic</a>
  3. Page-to-Page: the most common way to navigate from page-to-page is to create relative paths by creating relative URLs which work by creating a path to a page relative to the current page. This URL will be inserted in the nav tag between the quotes: href="here". The following three paths can be created:
    1. Same directory: The simplest way of navigating between pages is if the navigation occurs between the same directory. To do this we reference the page we are looking for and use that name as the relative URL. For example, if we have two files within the same directory--contacts.html and index.html--and we are attempting to create a path from contacts.html to index.html, we simply write: index.html
    2. Down the child directory: If we are in a directory and are attempting to access a page that is within a folder that is in the same directory as we are, the path we create is by accessing the desired folder in the directory and finally accessing the page within that folder. Our relative URL will begin with the folder's name, followed by "/" (which signifies the user accessing the folder), and finally the name of the page inside the folder. So if we are attempting to access page "index.html" that is inside folder “product”, a folder that is in the same directory as our current page, the URL would look like this: product/index.html
    3. Up the parent directory: If we are trying to navigate to a page that requires us to exit our folder into a parent directory to then enter a different folder to finally access our desired page, we must use the ".." symbol (which signifies exiting a folder, in contrast with "/" which signifies entering). Furthermore, if we have a directory with two folders: homepage and contacts--and we are inside the homepage folder in page index.html, and we are trying to navigate to contacts.html--which is found inside the contacts folder--we would write the following URL: ../contacts/contacts.html
